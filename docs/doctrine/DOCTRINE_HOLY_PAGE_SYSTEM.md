# Doctrine / Holy Page System

## Purpose
The Doctrine System is the campaign's governing truth layer.

It defines:
- what the campaign stands for
- how it speaks
- what it prioritizes
- what it will not do
- how decisions are judged
- how the campaign stays aligned over time

Doctrine is not a note bank or loose strategy memo.
Doctrine is active governing truth used by the entire office.

---

## Core doctrine structure

### 1. Core Doctrine
Non-negotiable campaign baseline:
- mission
- identity
- core beliefs
- core promises
- what Megan represents

### 2. Messaging Doctrine
- tone rules
- language style
- framing approach
- approved phrases
- disallowed language
- positioning strategy

### 3. Strategic Doctrine
- priority focus areas
- target audiences
- strategic lanes
- campaign posture

### 4. Operational Doctrine
- response standards
- quality expectations
- escalation philosophy
- speed vs depth rules

### 5. Temporary Doctrine
- campaign phase emphasis
- temporary strategic pushes
- short-term operational focus

---

## Core doctrine behaviors

### Always active
Every department and system should read doctrine before acting.

### Always enforced
Doctrine alignment should be checked in quality review and output generation.

### Always visible
Megan and the campaign manager should be able to inspect doctrine state, changes, conflicts, and active emphasis.

---

## Doctrine enforcement rules

### Quality Gate 1
Doctrine alignment must always be checked first.

If an output conflicts with doctrine:
- it must be flagged
- it must not silently pass
- it must be revised, escalated, or explicitly overridden

---

## Megan override rules

Megan can override doctrine at any time.

When Megan overrides doctrine, the system should ask:

- Is this a one-time exception?
- Is this a temporary override?
- Should doctrine be updated permanently?

### Override states
- one-time override
- temporary override
- permanent doctrine update

All overrides must be tracked.

---

## Doctrine change system

Doctrine should not change casually.

### Doctrine changes may originate from:
- Megan direct update
- Campaign Manager recommendation
- evidence-backed system recommendation

### A doctrine change recommendation must include:
- reason
- impact
- what changes
- risks
- examples
- recommendation confidence

---

## Doctrine conflict handling

Doctrine conflicts must be surfaced.

Examples:
- opportunity vs tone
- aggressiveness vs professionalism
- speed vs quality
- audience fit vs consistency

Required behavior:
1. detect conflict
2. present options
3. recommend direction
4. escalate if needed

---

## Doctrine versioning and memory

Doctrine must:
- version over time
- track what changed
- track why it changed
- preserve history
- support rollback if needed

---

## Doctrine + departments

All departments must use doctrine, but not identically.

Doctrine should be translated into department-specific behavior.

Examples:
- Communications uses doctrine for language and framing
- Field uses doctrine for conversation style and issue emphasis
- Fundraising uses doctrine for donor framing
- Social uses doctrine for channel-specific expression
- Strategy uses doctrine for campaign posture and prioritization

Doctrine should be translated, not copied blindly.

---

## Doctrine + Brainstorm / Development / Simulation

Brainstorm, Development, and Simulation must:
- use doctrine as a baseline
- challenge doctrine when appropriate
- test doctrine alternatives when useful
- propose evidence-backed improvements

---

## Billion-dollar refinement layer

Doctrine should not remain static and reactive.
It should become an active strategic intelligence system.

### 1. Doctrine Performance Layer
Doctrine should track:
- which messages perform
- which tones resonate
- which framing converts
- which audiences respond
- where doctrine is working or underperforming

### 2. Doctrine Tension Engine
The system should detect when:
- departments struggle to follow doctrine
- overrides happen often
- outputs repeatedly conflict with doctrine
- field or social signals suggest friction

### 3. Multi-Doctrine Testing
The system should be able to test alternate doctrine approaches:
- compare message approaches
- compare tone strategies
- compare framing across channels and audiences
- recommend stronger doctrine using evidence

### 4. Doctrine Confidence Score
Each doctrine element should have:
- confidence level
- evidence level
- stability rating
- performance support

### 5. Doctrine Scoping
Doctrine can vary by:
- audience
- channel
- situation
- campaign phase

Doctrine should not be one-size-fits-all when context matters.

### 6. Doctrine Priority Stack
Doctrine elements should have priority levels:
- Tier 1 non-negotiable
- Tier 2 strong preference
- Tier 3 flexible
- Tier 4 experimental

### 7. Doctrine-Driven Creation
Doctrine should not only validate outputs after the fact.
It should drive creation from the start.

### 8. Doctrine Alerts
The system should alert when:
- doctrine drift is increasing
- contradictions appear
- message inconsistency spreads
- performance of a doctrine element is weakening

### 9. Doctrine Simulation
Before major doctrine changes, the system should:
- simulate likely outcomes
- compare risks
- show tradeoffs
- recommend whether to update or preserve doctrine

### 10. Doctrine Learning Memory
Doctrine changes should remember:
- what triggered the change
- what changed
- what happened after the change
- whether the change improved outcomes

---

## Final doctrine principles

1. Doctrine is always active
2. Doctrine is always enforced
3. Doctrine can be overridden only by Megan
4. Doctrine can evolve with evidence
5. Doctrine conflicts must be surfaced
6. Doctrine is used by all departments
7. Doctrine is versioned and tracked
8. Doctrine drives creation, not just validation
9. Doctrine is measurable
10. Doctrine is testable
11. Doctrine is adaptive
12. Doctrine is continuously pressure-tested
